## What This Project Is {#overview}

A TypeScript + Astro sailing route guide. One Markdown file
(`src/content/route/amalfi.md`) is the single source of truth for all outputs:

- **Web page** (Astro → GitHub Pages at `https://sorokoletovdu.github.io/amalfi-route-2026`)
- **Printable PDF** (`@react-pdf/renderer`, pure Node.js, one page per day)

The page presents the Amalfi Coast route day-by-day: points of interest, destinations,
and anchoring information. Data is extended by an optional scraping script that enriches
POIs with ratings and data from TripAdvisor and Navily.

## Architectural Invariants {#invariants}

These constraints are non-negotiable. Any change that violates them requires an explicit
decision recorded in `WAL.md` and reported to the author:

- Web (Astro) and PDF (React-PDF) share **data**, never components.
- PDF is generated by **pure Node.js** (`@react-pdf/renderer`). No browser, no Playwright in CI.
- `src/content/route/amalfi.md` is the **single source of truth** for all outputs.
- Package manager is **pnpm** — never npm or yarn.
- TypeScript **strict mode** throughout (`"strict": true`, `"noUncheckedIndexedAccess": true`).
- Tailwind is v4 — no `tailwind.config.js`. CSS-only configuration via `@theme` in `global.css`.
- `Font.register()` in `src/pdf/fonts.ts` must use `path.resolve()` absolute paths — relative paths
  fail silently in CI, producing a PDF with fallback fonts.

## REVIEW Marker Protocol {#review-protocol}

When you encounter a spec ambiguity, a contradiction, or a decision you cannot resolve without
author input:

1. **Implement the spec as written** — do not invent alternatives or silently deviate.
2. **Add a REVIEW marker** at the relevant location in the file:
   ```
   <!-- REVIEW: [reason] -->
   ```
3. **Record the item** in `WAL.md` under `## Open REVIEW Items`.
4. **Report to the author** at the end of the session.
5. **Do NOT modify specs unilaterally.**

## WAL Structure {#wal-structure}

`WAL.md` must always contain exactly these sections in this order:

| Section | Anchor | Content |
|---|---|---|
| Header | — | `Last updated: YYYY-MM-DD \| Session: <description>` |
| SESSION OPEN PROCEDURE | — | Blockquote — 3 mandatory steps; never modify |
| Current State | `#state` | Area status table: one-line status per row |
| Pending Work | `#pending` | `[ ]` items with concrete commands or file paths |
| Open REVIEW Items | `#reviews` | One entry per unresolved `<!-- REVIEW: -->` marker |
| Do NOT Touch | `#dntouch` | Bullet list of paths AI must never modify directly |
| Known Issues | `#issues` | Numbered list of active blockers |
| Next Session Must Read | `#next` | Current in-progress state with exact file paths and commands |

**Size constraint**: WAL must stay under 3,000 tokens. Completed items collapse to one-line
summaries. Details belong in git history, not WAL.

**Pending Work rules**: every item must be actionable — include the exact command or the
exact file and what to do. Never write "investigate X" or "check Y".

**Next Session Must Read rules**: written for cold-start. No conditional `if working on X`
instructions. Write only what is unconditionally true right now.

---

## Session Close Procedure {#close}

Before ending a session, update `WAL.md`:

1. Update the `Last updated` line with today's date and a one-phrase session description.
2. In `## Current State`, update any area whose status changed.
3. In `## Pending Work`, check off completed items (`[x]`) and add new items this session.
4. In `## Open REVIEW Items`, add any new markers created this session.
5. In `## Next Session Must Read`, describe the current in-progress state with exact file paths.
   If nothing is in progress: "No work in progress — ask the user."
6. Keep WAL under 3,000 tokens.
